# Mindtrip 产品复现方案

> 本文档承接前文对 Mindtrip 网站的全局产品分析与功能拆解。  
> 本文档只从产品角度说明复现范围、核心功能、页面交互和 Agent 的产品职责。  
> 具体开发实现、接口、数据结构和 Skill 细节将在后续开发文档中展开。

---

## 1. 复现目标与范围

### 1.1 复现目标

本阶段要复现 Mindtrip 最核心的用户体验：

```text
用户用自然语言表达旅行需求
→ 系统理解用户意图
→ 系统推荐合适地点
→ 系统生成结构化 Trip Plan
→ 用户在行程工作台中查看地图、地点和推荐理由
→ 用户继续用自然语言或按钮进行局部修改
```

这不是复现一个单纯的聊天机器人，而是复现一个以 **Trip Plan** 为核心的 AI 旅行规划工作台。

用户最终获得的不是一段攻略文字，而是一个可以继续编辑、替换、保存和地图化查看的旅行计划。

### 1.2 当前优先复现的三个核心功能

基于前文分析，当前优先复现三个功能：

| 核心功能 | 为什么优先复现 |
|---|---|
| AI 对话规划 | 这是用户进入产品的主要入口，也是 Mindtrip 最容易被感知的核心体验 |
| Trip Plan 行程工作台 | 这是产品承载结果的核心对象，决定产品是否区别于普通 AI 聊天 |
| 推荐与排序系统 | 这是结果质量的关键，决定行程是否真实、合理、顺路、符合偏好 |

这三个功能形成最小核心闭环：

```text
对话输入
→ 推荐地点
→ 生成行程
→ 地图展示
→ 用户修改
→ 行程更新
```

只要这个闭环成立，产品就已经具备 Mindtrip 的核心形态。

### 1.3 当前暂不复现的功能

以下功能属于后续扩展，不放入当前阶段：

| 功能 | 暂不复现原因 |
|---|---|
| Collections | 需要地点保存、分组、分享体系，依赖核心 Trip Plan 先成立 |
| Google Pins | 需要外部地图导入和地点匹配，复杂度较高 |
| Start Anywhere | 涉及链接、图片、PDF、多模态内容识别，先不做 |
| Receipts / Confirmations | 涉及 OCR、订单结构化、时间轴冲突检测，独立复杂 |
| Flights | 需要航班数据、价格、实时搜索能力 |
| Events | 需要本地活动数据源 |
| Collaboration | 涉及多人权限、实时同步、群聊和投票 |
| Mobile Nearby | 更偏移动端定位场景，依赖位置上下文 |

这些功能不是不重要，而是它们都应该建立在核心 Trip Plan 成立之后。

### 1.4 复现成功的产品标准

当前阶段复现成功的标准是：

- 用户能从首页输入旅行需求。
- 系统能生成一个伦敦及周边五日游。
- 行程以分天、分时段、地点卡片的方式展示。
- 用户能看到每个地点的推荐理由。
- 用户能在地图上看到地点位置。
- 用户能继续修改某一天或某类地点。
- 系统能输出变更摘要。
- 修改后的行程仍然保持结构清晰、路线合理。

## 2. 核心功能复现方案

### 2.1 AI 对话规划

#### 功能定位

AI 对话规划是用户进入产品的主入口。

用户不需要先填写复杂表单，也不需要先理解目的地结构，只需要输入一段自然语言：

```text
帮我规划一个伦敦及周边五日游，喜欢博物馆、街区漫步、英式下午茶和周边小镇，不要太赶。
```

系统需要把这段话转化为一个可执行的旅行计划，而不是生成一段普通攻略文章。

#### 需要复现的能力

| 能力 | 产品说明 |
|---|---|
| 新建行程 | 根据目的地、天数、偏好、节奏生成 Trip Plan |
| 多轮对话 | 用户可以继续提出修改要求 |
| 局部修改 | 用户可以修改某一天、某个时段或某类地点 |
| 意图理解 | 系统能区分用户是在新建、修改、替换还是询问 |
| 偏好理解 | 系统能理解博物馆、街区漫步、下午茶、周边小镇、不要太赶等偏好 |
| 追问 | 当信息不足时，系统可以向用户追问 |
| 变更摘要 | 修改后告诉用户改了什么 |

#### 关键交互

新建行程：

```text
用户输入伦敦及周边五日游需求
→ 系统生成初版 Trip Plan
→ 展示分天行程、地点卡片、地图点位
```

局部修改：

```text
用户输入“第三天不要太多博物馆，换成更适合散步的安排”
→ 系统识别目标是 Day 3
→ 系统减少博物馆类地点
→ 系统增加街区漫步类地点
→ 系统输出变更摘要
```

#### 产品要求

AI 对话规划的输出必须进入 Trip Plan，而不是停留在聊天气泡里。

不合格输出：

```text
伦敦是一座历史悠久、文化丰富的城市，有许多值得探索的景点……
```

合格输出：

```text
已为你生成伦敦及周边五日游：
Day 1：Westminster 与 South Bank
Day 2：博物馆与西区
Day 3：Borough Market 与 East London
Day 4：Cambridge 一日游
Day 5：Windsor 或 Richmond 轻松收尾
```

---

### 2.2 Trip Plan 行程工作台

#### 功能定位

Trip Plan 行程工作台是产品的核心页面。

它承载 AI 生成的结果，也承载用户后续修改、替换、保存和地图查看。

如果只有聊天回复，没有 Trip Plan 工作台，产品会退化成普通 AI 问答。  
如果有 Trip Plan，用户就可以把 AI 输出当成一个旅行项目继续操作。

#### 需要复现的能力

| 能力 | 产品说明 |
|---|---|
| 分天展示 | 按 Day 1 到 Day 5 展示行程 |
| 时段展示 | 每天分为上午、下午、晚上或等价结构 |
| 地点卡片 | 展示地点名称、类型、区域、推荐理由 |
| 地图联动 | 点击行程定位地图，点击地图高亮行程项 |
| 替换地点 | 用户可以对不满意的地点点击 Replace |
| 删除地点 | 用户可以删除不想去的地点 |
| 添加地点 | 用户可以添加某类地点，如酒吧、下午茶、小镇 |
| 锁定地点 | 用户确认的地点不应被系统随意改掉 |
| 保存状态 | 用户刷新或返回时可以恢复当前 Trip |

#### 工作台建议结构

```text
┌─────────────────────────────────────────────┐
│ 顶部：Trip 标题 / 保存 / 分享占位            │
├──────────────┬────────────────┬─────────────┤
│ 对话区        │ 行程区          │ 地图区       │
│ 修改输入      │ Day 1 - Day 5   │ marker       │
│ 系统摘要      │ 地点卡片         │ 地点详情      │
└──────────────┴────────────────┴─────────────┘
```

#### 地点卡片内容

每个地点卡片至少需要展示：

| 字段 | 产品作用 |
|---|---|
| 地点名称 | 告诉用户去哪 |
| 地点类型 | 博物馆、街区、餐厅、下午茶、小镇等 |
| 所在区域 | 帮助用户判断路线是否合理 |
| 推荐理由 | 解释为什么适合当前用户 |
| 建议时段 | 上午、下午、晚上 |
| 预计停留时间 | 帮助判断行程节奏 |
| 操作按钮 | Replace、Delete、Lock、View on map |

#### 产品要求

Trip Plan 必须让用户感到“这是一个可以继续编辑的计划”，而不是一次性生成结果。

用户应该能很自然地做这些动作：

- 查看每天安排。
- 点击地点看地图。
- 替换某个地点。
- 要求某一天轻松一点。
- 删除不喜欢的地点。
- 保存当前行程。

---

### 2.3 推荐与排序系统

#### 功能定位

推荐与排序系统决定行程质量。

它负责回答：

```text
应该推荐哪些地点？
这些地点为什么适合用户？
哪些地点应该放在同一天？
哪些地点适合作为替换项？
如何避免行程太赶或太重复？
```

如果推荐与排序不可靠，即使界面像 Mindtrip，用户也会觉得结果像普通 AI 随机生成。

#### 需要复现的能力

| 能力 | 产品说明 |
|---|---|
| POI 推荐 | 根据目的地和偏好推荐地点 |
| 区域判断 | 将地点按 Westminster、South Bank、East London、Cambridge 等区域组织 |
| 类型匹配 | 匹配博物馆、下午茶、街区、小镇、市场、餐厅等 |
| 热门度判断 | 对第一次去伦敦的用户优先经典高价值地点 |
| 多样性控制 | 避免一天全是博物馆或全是市场 |
| 节奏匹配 | 根据“不要太赶”减少每天地点数量 |
| 替换候选 | 为某个地点提供 3-5 个替代选项 |
| 推荐理由 | 解释地点和用户偏好之间的关系 |

#### 伦敦五日游推荐逻辑示例

一个合理的伦敦及周边五日游可以按区域和体验组织：

| 天数 | 推荐方向 | 产品理由 |
|---|---|---|
| Day 1 | Westminster + South Bank | 适合初到伦敦，经典地标集中，路线直观 |
| Day 2 | British Museum / National Gallery / West End | 满足博物馆和文化偏好 |
| Day 3 | Borough Market + Shoreditch / East London | 街区漫步和餐饮体验更强 |
| Day 4 | Cambridge 或 Oxford | 周边小镇/大学城一日游 |
| Day 5 | Windsor / Richmond / Hampstead | 节奏轻松，适合收尾 |

#### 替换逻辑示例

用户说：

```text
我不太想去大英博物馆，有没有更小众一点的？
```

系统应该优先找：

- 同为文化/博物馆类型
- 不破坏当天路线
- 更小众或更轻量
- 适合当前行程节奏

可能候选：

- Sir John Soane's Museum
- Wallace Collection
- Courtauld Gallery
- Museum of London Docklands

#### 产品要求

推荐与排序系统不应该只追求“相关”，还要追求“可旅行”。

一个推荐是否合格，要同时满足：

- 用户感兴趣
- 地点真实
- 区域合理
- 节奏合理
- 类型多样
- 能解释原因

## 3. 页面与交互复现

### 3.1 首页 / Start Planning

#### 页面目标

让用户快速开始规划，而不是让用户先浏览大量营销内容。

#### 页面内容

- 产品一句话说明
- 大输入框
- 示例 prompt
- Start planning 按钮
- 热门目的地快捷入口

示例 prompt：

```text
帮我规划一个伦敦及周边五日游，喜欢博物馆、街区漫步、英式下午茶和周边小镇，不要太赶。
```

#### 用户操作

```text
用户输入旅行需求
→ 点击 Start planning
→ 系统进入 Trip Workspace
```

---

### 3.2 Trip Workspace

#### 页面目标

展示和编辑 Trip Plan。

这是用户主要停留的页面。

#### 页面区域

| 区域 | 作用 |
|---|---|
| Chat Panel | 用户继续提出修改和追问 |
| Itinerary Panel | 展示分天行程 |
| Map Panel | 展示地点位置和选中地点 |
| Recommendation Area | 展示替换候选和推荐卡片 |
| Change Summary | 展示本次修改摘要 |

#### 核心交互

生成行程：

```text
用户输入需求
→ 行程区出现五日行程
→ 地图区出现地点 marker
→ 对话区显示简短说明
```

点击地点：

```text
用户点击 Day 2 的某个地点
→ 地图定位到该地点
→ 显示地点详情
```

替换地点：

```text
用户点击 Replace
→ 推荐区展示候选地点
→ 用户选择一个
→ 行程和地图同步更新
```

局部修改：

```text
用户输入“Day 3 更适合散步一点”
→ 系统只更新 Day 3
→ Change Summary 展示变化
```

---

### 3.3 Place Card / Recommendation Card

#### 组件目标

让推荐结果可理解、可比较、可操作。

#### 卡片内容

- 地点名称
- 地点类型
- 所在区域
- 推荐理由
- 标签
- 建议停留时间
- View on map
- Replace / Add / Lock / Delete

#### 产品要求

每张卡片都要让用户知道三件事：

```text
这是什么？
为什么推荐给我？
我下一步能做什么？
```

---

### 3.4 Change Summary

#### 组件目标

让用户理解系统修改了什么。

#### 示例

```text
已调整 Day 3：
- 减少了一个博物馆安排
- 增加了 Shoreditch 街区漫步
- 保留 Borough Market 午餐
- 当天节奏更轻松，步行路线更集中
```

#### 产品要求

每次局部修改后都应该有变更摘要。  
否则用户会不知道系统到底改了哪里，也会降低信任感。

## 4. Agent 产品架构

这一章只从产品角度定义 Agent 的意义和职责，不展开具体 Skill。  
Skill 会在下一阶段单独拆。

### 4.1 OrchestratorAgent

#### 产品意义

OrchestratorAgent 是整个产品体验的调度者。

用户只看到一个自然的输入框和行程工作台，但系统背后需要判断当前请求应该走哪条路径。

它回答的问题是：

```text
用户是在新建行程，还是修改行程？
是否需要重新推荐地点？
是否只更新某一天？
是否需要追问？
```

#### 需要实现的功能

- 判断用户请求类型
- 读取当前 Trip 上下文
- 决定调用哪些 Agent
- 聚合最终结果
- 保存新的 Trip 状态
- 在信息不足时触发追问

---

### 4.2 IntentAgent

#### 产品意义

IntentAgent 负责理解用户到底想做什么。

它不是直接回答用户，而是把自然语言转成产品可以执行的结构化意图。

例如：

```text
第三天不要太多博物馆，换成更适合散步的安排。
```

应该被理解为：

```text
目标：Day 3
动作：减少博物馆
新增偏好：街区漫步
修改范围：局部修改
```

#### 需要实现的功能

- 识别新建、修改、替换、询问
- 提取目的地、天数、兴趣、节奏、预算
- 解析“第二天”“这个地点”“下午那段”等指代
- 判断是否需要追问
- 维护多轮对话上下文

---

### 4.3 RetrievalRankerAgent

#### 产品意义

RetrievalRankerAgent 负责让推荐结果可信。

它决定系统推荐哪些地点、如何排序、哪些地点适合同一天、哪些地点适合作为替换候选。

没有它，产品容易变成大模型凭空写攻略。

#### 需要实现的功能

- 根据目的地和兴趣找候选地点
- 根据区域过滤地点
- 根据用户偏好排序
- 根据热度和经典程度排序
- 控制同类地点不要过度集中
- 为替换操作提供候选地点
- 为 TripPlannerAgent 提供高质量候选列表

---

### 4.4 TripPlannerAgent

#### 产品意义

TripPlannerAgent 负责把推荐地点组织成真正的行程。

它决定：

```text
哪些地点放在同一天？
一天安排几个地点？
上午、下午、晚上怎么分？
周边小镇放哪一天？
怎么避免行程太赶？
```

这是 Trip Plan 是否可执行的关键。

#### 需要实现的功能

- 根据天数生成分天结构
- 按区域组织每天路线
- 根据节奏控制每天地点数量
- 分配上午、下午、晚上
- 平衡经典景点、餐饮、街区和周边游
- 支持局部修改某一天
- 尊重用户已确认或锁定的地点

---

### 4.5 ResponseComposerAgent

#### 产品意义

ResponseComposerAgent 负责把系统结果变成用户能理解、能继续操作的界面内容。

同样的行程数据，如果只是原始结构，用户难以理解；如果写成长文，又不方便编辑。

这个 Agent 的价值是把系统结果包装成清晰的产品反馈。

#### 需要实现的功能

- 生成简短自然语言说明
- 生成 Trip 标题和每日标题
- 生成推荐理由
- 生成修改摘要
- 生成地点卡片展示文案
- 生成地图展示所需信息
- 生成下一步操作建议

---

## 5. 多 Agent 产品工作流

### 5.1 新建行程流程

用户输入：

```text
帮我规划一个伦敦及周边五日游，喜欢博物馆、街区漫步、英式下午茶和周边小镇，不要太赶。
```

产品流程：

```text
用户输入
→ OrchestratorAgent 判断为新建行程
→ IntentAgent 提取目的地、天数、兴趣、节奏
→ RetrievalRankerAgent 找到伦敦及周边候选地点
→ TripPlannerAgent 生成五日行程结构
→ ResponseComposerAgent 生成用户可读说明、地点卡片和地图展示信息
→ Trip Workspace 展示结果
```

用户看到：

```text
Day 1：Westminster 与 South Bank
Day 2：博物馆与西区
Day 3：Borough Market 与 East London
Day 4：Cambridge 一日游
Day 5：Windsor / Richmond 轻松收尾
```

---

### 5.2 局部修改流程

用户输入：

```text
第三天不要太多博物馆，换成更适合散步的安排。
```

产品流程：

```text
用户输入
→ OrchestratorAgent 判断为修改行程
→ IntentAgent 定位目标 Day 3
→ RetrievalRankerAgent 找街区漫步和同区域候选地点
→ TripPlannerAgent 更新 Day 3
→ ResponseComposerAgent 生成变更摘要
→ Trip Workspace 更新 Day 3 和地图
```

用户看到：

```text
已调整 Day 3：
- 减少了一个室内博物馆安排
- 增加 Shoreditch 街区漫步
- 保留 Borough Market 午餐
- 当天节奏更轻松，路线更集中
```

---

### 5.3 替换地点流程

用户操作：

```text
点击 British Museum 卡片上的 Replace
```

产品流程：

```text
用户点击 Replace
→ OrchestratorAgent 判断为替换地点
→ RetrievalRankerAgent 查找同类型或同区域替代地点
→ ResponseComposerAgent 生成候选卡片
→ 用户选择其中一个
→ TripPlannerAgent 更新行程
→ Trip Workspace 更新地点和地图
```

候选示例：

```text
1. Sir John Soane's Museum
2. Wallace Collection
3. Courtauld Gallery
4. Museum of London Docklands
```

---

### 5.4 追问流程

用户输入：

```text
帮我安排一个五日游。
```

产品流程：

```text
用户输入
→ IntentAgent 发现缺少目的地
→ OrchestratorAgent 不继续生成行程
→ ResponseComposerAgent 生成追问
```

用户看到：

```text
你想去哪个城市或地区？比如伦敦及周边、巴黎、京都等。
```

### 5.5 保存与恢复流程

用户操作：

```text
点击 Save，或刷新页面后回到当前 Trip
```

产品流程：

```text
Trip Workspace 当前状态
→ OrchestratorAgent 保存 Trip 状态
→ 用户再次进入页面
→ 系统恢复 Trip Plan、对话上下文和地图选中状态
```

用户感知：

```text
这不是一次性 AI 回复，而是一个可以继续编辑的旅行计划。
```
